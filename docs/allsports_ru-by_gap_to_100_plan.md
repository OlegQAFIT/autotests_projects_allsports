# Gap-to-100 Plan: Allsports Public Website (`/ru-by`)

## 1. Цель
Закрыть ключевые пробелы автотестов так, чтобы можно было обоснованно говорить о максимальном покрытии функциональности публичного сайта `https://www.allsports.by/ru-by` для релизной уверенности.

Важно: абсолютные "100%" в вебе недостижимы из-за динамики контента и внешних интеграций. Цель этого плана: 95%+ критичного покрытия с прозрачными остаточными рисками.

## 2. Текущее состояние (база)
- Основные страницы из sitemap покрыты UI тестами.
- Есть smoke и regression.
- Есть проверки форм, валидаций, документов, ссылок, endpoint-ов и консоли.
- Покрыт функционал `facilities` и `facilities-table` (включая фильтры/поиск).

## 3. Главные пробелы
1. Нет отдельного полноценного тест-пакета для страницы `/ru-by/app` (страница существует и доступна).
2. Нет обязательного регулярного прогона матрицы браузеров (`chrome + firefox + edge`) по одному и тому же набору.
3. Нет полного автоматического обхода всех внутренних ссылок на сайте с проверкой целевых URL и статусов.
4. Нет визуальной регрессии (скриншотных baseline-сравнений) для критичных блоков.
5. Нет accessibility-проверок (a11y) как обязательного quality gate.
6. Нет расширенного performance gate (кроме единичных проверок времени загрузки).
7. Нет отдельного набора для негативных API-кейсов (валидности payload, лимитов, антиспама, кодов ошибок).

## 4. Приоритетный backlog (по этапам)

### Этап A (критичный, 1-2 недели)
1. Добавить полноценный пакет `test_new_web_site/test_app_page.py` для `/ru-by/app`.
2. Добавить динамический crawler-тест внутренних ссылок:
   - собрать все `a[href]` с ключевых страниц;
   - отфильтровать `mailto/tel/#/javascript`;
   - для внутренних ссылок проверить HTTP `200/3xx` и отсутствие `404/5xx`;
   - проверять корректность canonical/финального URL.
3. Усилить `facilities-table`:
   - отдельные тесты каждого фильтра по значениям;
   - комбинации 2-3 фильтров;
   - проверка "сбросить фильтры" на возврат baseline набора.
4. Вынести набор "release blocker" в единый маркер (`-m release_gate`), чтобы запускался стабильно и быстро.

### Этап B (высокий, 2-3 недели)
1. Кроссбраузерная матрица в CI:
   - `chrome` обязательно;
   - `firefox` и `edge` как параллельные job;
   - минимум smoke + release_gate в каждом браузере.
2. Визуальная регрессия:
   - baseline скриншоты для ключевых секций (main, facilities, companies, partners, contacts, levels, footer, header);
   - допустимый порог diff;
   - отчет по артефактам в CI.
3. A11y-пакет:
   - автоматическая проверка базовых нарушений (контраст, aria-label, heading order, form labels);
   - критичные/major нарушения блокируют релиз.

### Этап C (средний, 2-4 недели)
1. Performance suite:
   - метрики `LCP`, `CLS`, `INP` для главных страниц;
   - пороги по desktop/mobile;
   - история тренда по прогонам.
2. API negative suite:
   - пустые/битые payload;
   - неверные форматы email/phone;
   - дубли/rate limit;
   - проверка корректных сообщений ошибок и кодов.
3. Контент-контроль:
   - словарь обязательных копирайт-блоков;
   - проверка отсутствия "битых" плейсхолдеров и техтекста.

## 6. CI/CD: обязательные quality gates
1. `smoke` (быстрый) на каждый PR.
2. `release_gate` на merge в `main`.
3. `regression` nightly.
4. `cross-browser smoke/release_gate` nightly или перед релизом.
5. `a11y + link-crawler + console` обязательны для релиза.

## 7. Критерии "достаточного покрытия"
Считаем, что покрытие близко к целевому, когда одновременно выполняются условия:
1. Все тесты `release_gate` зеленые.
2. Нет `SEVERE` ошибок в консоли на ключевых страницах.
3. Все внутренние ссылки сайта валидны (без `404/5xx`).
4. Формы проходят позитив/негатив сценарии в UI и API.
5. Кроссбраузерный smoke зеленый минимум в `chrome/firefox/edge`.
6. A11y критичные нарушения = 0.
7. Визуальные diff выше порога отсутствуют.

## 8. Порядок внедрения (конкретно)
1. Добавить `test_app_page.py` + link crawler.
2. Расширить `facilities-table` на granular-фильтры.
3. Ввести `release_gate` marker и отдельный CI job.
4. Подключить cross-browser matrix.
5. Добавить a11y suite.
6. Добавить visual regression.
7. Добавить performance suite.

## 9. Риски, которые останутся даже после внедрения
- Изменения сторонних сервисов (Mapbox, AppStore/GooglePlay, внешние CDN).
- Флаки из-за сетевой нестабильности и rate-limits.
- Контентные изменения без предварительного уведомления (маркетинг/копирайт).

## 10. Рекомендуемый формат отчетности после каждого релизного прогона
- `Total/Passed/Failed/Skipped`
- Failed by category: UI / API / Links / Console / A11y / Visual / Performance
- Список критичных дефектов (если есть)
- Итоговый статус: `GO` / `GO with risk` / `NO-GO`
